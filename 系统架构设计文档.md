### 3.1-补充 等级体系设计（参考B站升级规则）

#### 升级经验规则

| 等级 | 等级名称 | 升至该级所需累计经验 | 每完成1单获得基础经验 | 手续费折扣率 |
|------|---------|---------------------|----------------------|-------------|
| Lv.1 | 新手接单者 | 0（默认） | — | 无折扣（×1.00） |
| Lv.2 | 初级接单者 | 200 exp | — | 九五折（×0.95） |
| Lv.3 | 熟练接单者 | 1500 exp | — | 九折（×0.90） |
| Lv.4 | 资深接单者 | 4500 exp | — | 八五折（×0.85） |
| Lv.5 | 专业接单者 | 10800 exp | — | 八折（×0.80） |
| Lv.6 | 顶级接单者 | 28800 exp | — | 七五折（×0.75） |

> 说明：折扣作用于**平台手续费**，不影响任务金额。例如：任务金额100元，基础手续费5%=5元，Lv.3接单者实际手续费=5×0.90=4.5元，实际到账95.5元。

#### 经验获取规则

| 行为 | 获得经验 | 说明 |
|------|---------|------|
| 完成任务（结算成功） | 任务金额×2（取整） | 50元任务得100 exp |
| 获得5星好评 | +10 exp | 额外奖励 |
| 获得4星好评 | +5 exp | 额外奖励 |
| 违规记录（超时/取消） | -20 exp | 惩罚，最低不低于当前等级下限 |

#### 升级检测时机

- 每次结算成功后，Service层自动计算新 `exp`，判断是否触发升级（`level` +1）
- 升级时推送系统消息：「恭喜！你已升至 Lv.{n}，手续费折扣提升为×{rate}」
- 降级规则：**不降级**（与B站一致，等级只升不降，违规仅扣经验值）

### 3.2 任务表 `task`

任务状态流转：`待审核(0)` → `待接单(1)` → `已抢单-待协商(2)` → `待支付(3)` → `进行中(4)` → `已完成(5)` → `已结算(6)` → `已互评(7)` / `已取消(8)`

```sql
CREATE TABLE `task` (
  `id`              BIGINT        NOT NULL COMMENT '任务ID（雪花算法）',
  `publisher_id`    BIGINT        NOT NULL COMMENT '发布者ID',
  `acceptor_id`     BIGINT                 COMMENT '当前抢单者ID（锁单中/进行中）',
  `title`           VARCHAR(100)  NOT NULL COMMENT '任务标题',
  `category`        TINYINT       NOT NULL COMMENT '分类：1代取快递 2资料整理 3编程 4代课占位 5其他(我后面加了一个增加标签的功能实际上可能还会更多category)',
  `description`     TEXT          NOT NULL COMMENT '任务描述',
  `amount`          DECIMAL(10,2) NOT NULL COMMENT '任务金额（1-500元）',
  `fee_amount`      DECIMAL(10,2)           COMMENT '实际手续费（结算时写入）',
  `delivery_type`   TINYINT       NOT NULL DEFAULT 0 COMMENT '交付方式：0线上 1线下',
  `deadline`        DATETIME      NOT NULL COMMENT '截止时间',
  `status`          TINYINT       NOT NULL DEFAULT 0 COMMENT '任务状态：0待审核 1待接单 2已抢单待协商 3待支付 4进行中 5已完成 6已结算 7已互评 8已取消',
  `need_audit`      TINYINT       NOT NULL DEFAULT 0 COMMENT '是否需要审核（金额>200自动置1）',
  `task_images`     JSON                    COMMENT '任务图片列表（最多9张，URL数组）',
  `delivery_proof`  JSON                    COMMENT '交付成果文件列表（图片/文件URL数组，最多9张）',
  `reject_reason`   VARCHAR(200)            COMMENT '验收不通过原因',
  `lock_expire_at`  DATETIME                COMMENT '锁单过期时间（抢单后+20分钟，超时自动释放）',
  `created_at`      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at`      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  INDEX `idx_publisher` (`publisher_id`),
```

### 3.4 结算记录表 `settlement`

```sql
CREATE TABLE `settlement` (
  `id`            BIGINT        NOT NULL COMMENT '结算ID',
  `task_id`       BIGINT        NOT NULL UNIQUE COMMENT '关联任务ID',
  `acceptor_id`   BIGINT        NOT NULL COMMENT '承接者ID',
  `task_amount`   DECIMAL(10,2) NOT NULL COMMENT '任务金额',
  `base_fee_rate` DECIMAL(5,4)  NOT NULL COMMENT '基础手续费率快照（来自fee_config）',
  `level_at_settle` TINYINT     NOT NULL COMMENT '结算时接单者等级快照（1-6）',
  `fee_discount`  DECIMAL(5,4)  NOT NULL COMMENT '等级折扣率快照（如0.90=九折）',
  `fee_rate`      DECIMAL(5,4)  NOT NULL COMMENT '实际手续费率=base_fee_rate×fee_discount',
  `fee_amount`    DECIMAL(10,2) NOT NULL COMMENT '实际手续费金额',
  `real_amount`   DECIMAL(10,2) NOT NULL COMMENT '实际到账金额=task_amount-fee_amount',
  `exp_gained`    INT           NOT NULL DEFAULT 0 COMMENT '本次结算获得的经验值',
  `status`        TINYINT       NOT NULL DEFAULT 0 COMMENT '0已结算 1已提现',
  `settled_at`    DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='结算记录表';
```

**结算计算逻辑**：

```
1. 查询 fee_config 获取基础手续费率 base_fee_rate（按任务金额阶梯匹配）
2. 查询接单者当前 level，映射折扣率 fee_discount（见等级表）
3. 实际手续费率 = base_fee_rate × fee_discount
4. 手续费金额   = task_amount × 实际手续费率（BigDecimal，HALF_UP保留2位）
5. 实际到账     = task_amount - fee_amount
6. 更新接单者余额：balance += real_amount
7. 更新接单者总收入：total_earned += real_amount（只增不减）
8. 计算经验值：exp_gained = floor(task_amount × 2)
9. 更新接单者经验：exp += exp_gained
10. 判断是否升级：while (exp >= 当前等级升级所需经验) { level++ }
11. 若升级：推送系统消息给接单者
12. 写入 settlement 记录（含等级/折扣快照）
```

**抢单前置校验逻辑**（`POST /api/tasks/{id}/grab`）：

```
1. 查 grab_record：当日该用户抢单总次数 ≥ 5 → 拒绝，提示「今日抢单次数已达上限」
2. 查 grab_record：当日该用户对该任务已有记录 → 拒绝，提示「今日已抢过该任务」
3. 查 user_violation：当日违规次数 ≥ 3 → 拒绝，提示「今日违规次数过多，明日可继续」
4. 查 user.credit_score（信誉分用作押金）：信誉分 < 1 → 拒绝，提示「积分不足，无法抢单」
5. 全部通过 → 冻结1信誉分（credit_score-1），创建 grab_record，更新 task 状态为2，
              设置 lock_expire_at = NOW() + 20分钟，
              推送系统消息给发布者
```

**超时定时任务**（每分钟扫描一次）：

```
查询：task.status = 2 AND lock_expire_at ≤ NOW()
执行：
  → task.status 改为 1（待接单），清空 acceptor_id、lock_expire_at
  → 扣除接单者冻结信誉分（不返还，credit_score 已在抢单时扣除）
  → 额外扣20经验值（不低于当前等级下限）
  → grab_record.status 改为 3（超时释放），is_violation = 1
  → user_violation 插入1条记录（类型3：协商超时）
  → 推送消息给双方
```

## 六、Redis 缓存策略

| Key格式 | 缓存内容 | TTL | 刷新时机 |
|---------|---------|-----|----------|
| `token:{userId}` | JWT Token（用于主动登出失效） | 7天 | 登录时写入，登出时删除 |
| `sms:code:{phone}` | 短信验证码 | 5分钟 | 每次发送覆盖 |
| `task:list:hot` | 热门任务列表（待接单状态） | 5分钟 | 定时刷新/任务状态变更时删除 |
| `task:detail:{id}` | 任务详情 | 10分钟 | 任务状态变更时删除 |
| `fee:config` | 手续费配置列表 | 永久（手动） | 管理员更新配置时立即刷新 |
| `user:credit:{id}` | 用户信誉分 | 30分钟 | 评价提交后删除 |
| `user:unread:{id}` | 未读消息数 | 30分钟 | 消息变更时更新 |
| `grab:daily:{userId}:{date}` | 用户当日抢单次数 | 至当日23:59 | 每次抢单+1 |
| `grab:task:{taskId}:{userId}:{date}` | 用户当日对某任务是否已抢过 | 至当日23:59 | 抢单时写入 |
| `violation:daily:{userId}:{date}` | 用户当日违规次数 | 至当日23:59 | 记违规时+1 |
| `user:level:{userId}` | 用户等级和经验值 | 1小时 | 结算升级后删除刷新 |

**动态手续费核心实现**：管理员调用 `PUT /api/admin/fee-config` 后，`FeeConfigServiceImpl` 同步更新数据库并调用 `redisTemplate.delete("fee:config")` 刷新缓存，下次计算手续费时重新从数据库加载并写入缓存，**无需重启服务即可生效**（比赛技术亮点）。

## 七、安全设计

### 7.1 认证流程

```
客户端登录 → AuthController.login()
    → 校验密码（SHA256+盐）
    → 生成JWT（payload含userId、role、exp）
    → 写入Redis: token:{userId} = JWT，TTL 7天
    → 返回Token给前端

后续请求 → JwtAuthFilter
    → 解析Header: Authorization: Bearer <token>
    → 验证JWT签名及有效期
    → 比对Redis中的Token（防止登出后重用）
    → 注入SecurityContext，后续@PreAuthorize生效
```

### 7.2 权限分层

```
公开接口（无需Token）：
  /api/auth/**

普通用户接口（需要Token，角色0或1）：
  /api/user/**
  /api/tasks/**
  /api/payment/**
  /api/settlement/**
  /api/review/**
  /api/message/**

管理员接口（需要Token + role=1）：
  /api/admin/**
  → 使用 @PreAuthorize("hasRole('ADMIN')") 注解保护
```

### 7.3 密码安全

- 注册时生成16位随机盐值，存储 `SHA256(password + salt)`
- 传输层密码由前端进行一次MD5处理后再传输，防止明文泄露

### 7.4 接口安全

- 所有接口参数使用 `@Valid` + JSR-303注解校验
- 金额相关操作使用数据库事务（`@Transactional`）保证一致性
- 防止越权：Service层校验操作者身份（如只有任务发布者才能验收）

## 十一、比赛演示脚本（核心流程）

1. **注册登录**：用学号注册账号A（发布者）、账号B（接单者）
2. **发布任务**：账号A发布「编程」类任务，金额50元，上传任务图片，提交后任务进入「待接单」状态
3. **抢单**：账号B点击「抢单」，系统校验通过后冻结1积分，任务进入「已抢单-待协商」，双方可查看联系方式
4. **协商确认**：账号A收到系统消息，沟通后点击「接受接单者」，任务变为「待支付」
5. **付款托管**：账号A支付50元至平台托管，任务变为「进行中」，账号B积分返还
6. **提交成果**：账号B完成任务，上传交付凭证图片（最多9张）
7. **验收结算**：账号A验收通过，平台自动扣除手续费（账号B为Lv.1无折扣，5%=2.5元），账号B到账47.5元，获得100 exp
8. **等级演示**：查看账号B个人主页，展示等级徽章Lv.1、经验进度条100/200、历史总收入¥47.50
9. **动态手续费**：管理员登录后台，将手续费调整为8%，**立即生效无需重启**（Redis缓存亮点）
10. **互评**：双方评价，展示信誉分更新
11. **防恶意演示**：展示账号B当日抢单3次后被限制，或违规3次当日禁止抢单
12. **管理后台**：展示用户管理、任务审核、违规记录、交易统计功能
