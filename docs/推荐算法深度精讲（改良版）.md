## 2. 特征工程：从文字到向量 {#特征工程}

特征工程是推荐系统的地基。**垃圾进，垃圾出**——如果标签提取得很差，后面的打分再精细也没用。

### 2.1 旧方案：正则分词的缺陷

```java
// 旧代码
src.split("[\\s\\p{Punct}\\u3000-\\u303F]+")
```

问题：

| 输入文本 | 旧方案输出 | 问题 |
|---------|-----------|------|
| 「帮我整理一下Excel数据」 | [帮我, 整理, 一下, Excel, 数据] | 「帮我」「一下」是噪声 |
| 「需要Python编程高手」 | [需要, Python, 编程, 高手] | 「需要」是停用词 |
| 「20元跑腿取快递」 | [20元, 跑腿, 取, 快递] | 「20元」是无意义数字 |

### 2.2 新方案：HanLP 名词提取

```java
// TaskTagExtractor.java 核心逻辑
List<Term> terms = HanLP.segment(text);
return terms.stream()
    .filter(t -> t.nature != null)
    .filter(t -> {
        String nat = t.nature.toString();
        return nat.startsWith("n") && !nat.startsWith("nr"); // 名词，排除人名
    })
    .map(t -> t.word.trim().toLowerCase())
    .filter(w -> w.length() >= 2)        // 去除单字
    .filter(w -> !isPureNumber(w))       // 去除纯数字
    .filter(w -> !EXTRA_STOPWORDS.contains(w))  // 去除业务停用词
    .distinct()
    .limit(20)
    .collect(Collectors.toList());
```

**HanLP 词性标注体系**（你需要记住）：

| 词性代码 | 含义 | 是否保留 |
|---------|------|--------|
| `n` | 普通名词（数据、编程、设计）| ✅ 保留 |
| `nn` | 专有名词（Python、Excel）| ✅ 保留 |
| `nz` | 其他专名（特定领域词）| ✅ 保留 |
| `nt` | 机构团体名 | ✅ 保留 |
| `ns` | 地名 | ✅ 保留 |
| `nr` | 人名 | ❌ 排除（避免人名偏移推荐） |
| `v` | 动词（帮忙、完成）| ❌ 排除 |
| `d` | 副词（很、非常）| ❌ 排除 |
| `p` | 介词（对、在）| ❌ 排除 |

**效果对比**：


## 3. 五维打分模型（含改良细节）{#五维打分}

### 3.1 总公式

$$
\text{score} = \underbrace{\sum_{k \in \{a,c,m,d,v\}} w_k \cdot s_k}_{\text{五维主分}} + \underbrace{B_u + B_c + B_l}_{\text{三项奖励}}
$$

最终 clamp 到 $[0, 1]$。

### 3.2 维度一：能力标签（ability）—— Jaccard 相似度

**数学定义**：

$$
J(A, B) = \frac{|A \cap B|}{|A \cup B|}, \quad s_{ability} = 0.20 + 0.80 \times J(A, B)
$$

其中 $A$ = 任务标签集合（HanLP 提取），$B$ = 用户能力标签集合。

**为什么映射到 [0.20, 1.00] 而不是 [0, 1]？**

这是一个**基础分设计**的重要思路：
- 若所有维度都从 0 开始，那么完全不匹配的任务得分 = 0
- 一旦该维度权重大，0 分会严重拉低总分，导致区分度过强
- 给一个 0.20 的基础分，表示「即使没有标签匹配，任务本身还是有一定价值的」

```
用户标签：{Python, 数据分析}

任务A 标签：{Python, 机器学习} → 交集={Python}, 并集={Python,数据分析,机器学习}
  J = 1/3 ≈ 0.333 → s_ability = 0.20 + 0.80×0.333 = 0.467

任务B 标签：{PPT, Word}       → 交集={}, 并集={Python,数据分析,PPT,Word}
  J = 0/4 = 0      → s_ability = 0.20 + 0.80×0    = 0.200

任务C 标签：{Python, 数据分析} → 交集={Python,数据分析}, 并集={Python,数据分析}
  J = 2/2 = 1.0    → s_ability = 0.20 + 0.80×1.0  = 1.000
```

**改良要点**：标签在计算前统一小写（`.toLowerCase()`）并过滤停用词，
确保「Python」和「python」视为同一个词，避免大小写导致的漏匹配。

### 3.3 维度二：分类偏好（category）—— 
软惩罚二值映射

| 情况 | 得分 | 设计原因 |
|------|------|---------|
| 命中偏好分类 | 1.00 | 强烈偏好信号 |
| 未命中偏好分类 | **0.20**（原 0.35）| 软惩罚，避免非偏好任务被完全屏蔽 |
| 用户未设置偏好 | 0.50 | 中性，不影响排序 |

**为什么从 0.35 改为 0.20？**

原来 0.35 的惩罚力度不够，非偏好分类任务仍能排在前列。以 ability=0.90、amount=0.90 的任务为例：

```
旧版（category未命中=0.35）：
  主分 = 0.90×0.30 + 0.35×0.25 + 0.90×0.20 ≈ 0.625  → 仍排名很高！

新版（category未命中=0.20）：
  主分 = 0.90×0.30 + 0.20×0.25 + 0.90×0.20 ≈ 0.580  → 明显更低，区分度增强
```

### 3.4 维度三：金额区间（amount）—— 梯形函数

$$
s_{amount} = \begin{cases}
1.0 & lo \leq amt \leq hi \\
\max\left(0.10,\ 1 - \dfrac{lo - amt}{span}\right) & amt < lo \\
\max\left(0.10,\ 1 - \dfrac{amt - hi}{span}\right) & amt > hi
\end{cases}
$$

其中 $span = hi - lo$，若 $span < 10$ 则兜底用 $span = 50$。

**关键改良：衰减基准换成区间宽度**

```
用户金额区间 [5, 55]，span = 50

任务金额 = 0 元

❌ 旧错误：decay = (5-0)/5 = 1.0  → 得分=0（只差5元却给0分，不合理）
✅ 新正确：decay = (5-0)/50 = 0.1 → 得分=0.90（合理，偏离区间宽度10%）
```

### 3.5 维度四：距离（distance）—— 分段线性

$$
s_{distance} = \begin{cases}
1.0 & \text{线上任务或 dist=0} \\
1.0 - 0.4 \times \dfrac{dist}{maxDist} & 0 < dist \leq maxDist \\
\max\left(0.10,\ 0.60 - 0.50 \times \dfrac{dist - maxDist}{maxDist}\right) & dist > maxDist
\end{cases}
$$

```
用户 maxDist = 10km：
  dist=0km  → 1.00
  dist=5km  → 1.0 - 0.4×0.5  = 0.80
  dist=10km → 1.0 - 0.4×1.0  = 0.60（边界）
  dist=15km → max(0.10, 0.60-0.50×0.5) = 0.35
  dist=20km → max(0.10, 0.60-0.50×1.0) = 0.10
```

### 3.6 维度五：交付方式（delivery）—— 非对称二值

| 情况 | 得分 | 改良说明 |
|------|------|--------|
| 命中偏好 | 1.00 | 不变 |
| 未命中偏好 | **0.40**（原 0.50）| 差距从 0.50 扩大到 0.60，信号更强 |
| 用户未设置 | 0.70 | 不变 |

## 4. 三项奖励加成 {#三项奖励}

### 4.1 时效性奖励（urgencyBonus） [0, 0.10]

三段式设计，h = 距截止剩余小时数：

```
h 在 (0, 24]    → bonus = 0.07 + 0.03×(1 - h/24)    范围 [0.07, 0.10]
h 在 (24, 72]   → bonus = max(0.02, 0.07-0.05×((h-24)/48))  范围 [0.02, 0.07)
h 在 (72, 168]  → bonus = max(0, 0.02×(1-(h-72)/96))    范围 [0, 0.02)
h > 168 或 ≤0  → bonus = 0
```

**改良关键**：原版 h=0（已过期）反而得最高分，逻辑方向错误，已修正。

**工程细节**：用 `ChronoUnit.HOURS.between(now, deadline)` 计算，时区统一 `Asia/Shanghai`。

### 4.2 发布者信誉奖励（creditBonus） [0, 0.05]

$$
B_c = 0.05 \times \min\left(1,\ \frac{\max(0,\ credit - 100)}{100}\right)
$$

从预加载的 publisherCache（Map）取，**零额外查询**。

### 4.3 等级匹配奖励（levelBonus） [0, 0.05]

| 等级 | 舒适金额区间 |
|------|------------|
| 1 | [0, 80) |
| 2 | [80, 150) |
| 3 | [150, 250) |
| 4 | [250, 400) |
| 5 | [400, 600) |
| 6 | [600, +∞) |

金额落在舒适区间内 → 0.05 满分；偏离越远奖励越少，最低 0。

## 5. 权重自适应：梯度下降的工程实践 {#权重自适应}

### 5.1 核心公式

$$
w_k^{new} = \text{clamp}\left(w_k^{old} + \eta \cdot g_k,\ [0.05,\ 0.55]\right)
$$

- $\eta = 0.04$（结算后学习率）
- clamp 防止权重趋向 0 或 1（太极端会让其他维度失效）
- 更新后必须 normalize，使所有权重之和 = 1

### 5.2 梯度信号计算

**连续型维度**（ability / amount）：

$$
g_k = (s_k - 0.5) \times 2 \in [-1, +1]
$$

子分 > 0.5 → 正梯度（这个维度在这次接单中表现好，增大权重）  
子分 < 0.5 → 负梯度（表现差，减小权重）

**二值型维度**（category / delivery）——非对称梯度：

| 维度 | 命中 | 未命中 | 非对称原因 |
|------|------|--------|----------|
| category | +1.00 | -0.60 | 用户可能偶尔接非偏好任务，不应过度惩罚 |
| delivery | +0.80 | -0.40 | 同上，弱化未命中的负向学习 |

**一次完整更新示例**：

```
用户接了一个 Python 编程任务（category命中，amount偏高，distance=0）

初始权重：ability=0.30, category=0.25, amount=0.20, distance=0.15, delivery=0.10

子分：
  ability  = 0.85 → g = (0.85-0.5)×2 = +0.70
  category = 1.00 → g = +1.00（命中）
  amount   = 0.35 → g = (0.35-0.5)×2 = -0.30
  distance = 1.00 → g = (1.00-0.5)×2 = +1.00
  delivery = 0.40 → g = -0.40（未命中，非对称）

更新（η=0.04）：
  ability  : 0.30 + 0.04×0.70 = 0.328
  category : 0.25 + 0.04×1.00 = 0.290
  amount   : 0.20 + 0.04×(-0.30) = 0.188  ← 减小
  distance : 0.15 + 0.04×1.00 = 0.190
  delivery : 0.10 + 0.04×(-0.40) = 0.084  ← 减小

归一化后：ability↑, category↑, amount↓, distance↑, delivery↓
结论：系统学到「这个用户更在意技能和分类，不太在意金额和交付方式」
```

### 5.3 归一化（Normalize）

```java
private Map<String, BigDecimal> normalize(Map<String, BigDecimal> raw) {
    BigDecimal sum = raw.values().stream().reduce(BigDecimal.ZERO, BigDecimal::add);
    // 最后一个 key 用 1-acc 补齐，消除浮点累积误差
    List<String> keys = new ArrayList<>(raw.keySet());
    BigDecimal acc = BigDecimal.ZERO;
    for (int i = 0; i < keys.size() - 1; i++) {
        BigDecimal v = raw.get(keys.get(i)).divide(sum, 6, RoundingMode.HALF_UP);
        result.put(keys.get(i), v);
        acc = acc.add(v);
    }
    result.put(lastKey, BigDecimal.ONE.subtract(acc).max(BigDecimal.ZERO));
}
```

**工程细节**：最后一个 key 用 `1 - acc` 填充而不是 `raw[k]/sum`，
这样能保证所有权重之和**精确等于 1**，消除 BigDecimal 除法的累积误差。

## 6. 点击行为反馈：新增的轻量学习信号 {#点击反馈}

### 6.1 
为什么需要点击反馈？

结算后才更新权重，反馈周期太长。点击行为是更即时的信号：用户点开详情，说明他对这个任务有兴趣。

### 6.2 学习率对比

| 触发时机 | 学习率 | 信号强度 |
|---------|--------|------|
| 结算后 | 0.04 | 强（真实完成）|
| 点击详情 | **0.01** | 弱（仅表示兴趣）|

学习率 0.01 = 结算的 1/4，避免随手点开就大幅修改权重。

### 6.3 信号累积效果

```
用户连续点击 5 个 Python 编程任务（s_ability≈0.85）：

ability 权重：0.300 → 0.307 → 0.314 → 0.320 → 0.326 → 0.333

→ 下次推荐时编程类任务排名更靠前
```

前端在 TaskDetail.vue 的 onMounted 中调用，`.catch(()=>{})` 保证失败不影响主流程。

## 7. 协同过滤召回 {#协同过滤}

### 7.1 基于内容 vs 协同过滤

**基于内容**（主路）：分析任务属性与画像匹配。画像不准时效果差。

**协同过滤**（副路）：完全不看内容，只看行为相似性。

> "和你行为相似的用户都接了任务A，但你还没接过" → 推荐A

### 7.2 余弦相似度

$$
\text{sim}(u, v) = \frac{|u \cap v|}{\sqrt{|u|} \times \sqrt{|v|}}
$$

```
用户A：{任务1,2,4}  用户B：{任务1,3,4}
交集：{任务1,4}，|交集|=2
sim = 2/(√3×√3) = 0.667
```

### 7.3 召回流程

```
1. 拉当前用户历史接单 taskId 集合
2. 拉最近500条其他用户记录，按userId分组
3. 计算余弦相似度，找 Top-5 相似用户
4. 召回相似用户接过但当前用户未接过的任务
   CF得分 = Σ sim(我, v) for all 相似用户v who grabbed this task
5. 查任务详情 → receivable过滤 → 归一化 → Top-K
```

新用户 myTaskIds 为空时直接返回空，主路自动兜底。

## 8. 融合策略 {#融合策略}

$$
\text{score}_{fused} = \begin{cases}
0.7 \times \text{score}_{main} + 0.3 \times \text{score}_{cf} & \text{两路均有} \\
\text{score}_{main} & \text{仅主路} \\
0.3 \times \text{score}_{cf} & \text{仅CF路（补充多样性）}
\end{cases}
$$

CF独有任务分数上限≈0.3，不会超越主路高分任务，只在主路不够时补充。两路重叠的任务推荐理由自动追加「· 相似用户也接过」。

## 11. 性能优化：消灭N+1查询 {#性能优化}

**问题**：300条候选任务，每条查一次发布者 = 300次数据库查询。

**解决**：batchLoadPublishers() 一次IN查询加载全部发布者，从Map O(1)取。

300次查询 → **1次查询**，是推荐系统最重要的工程优化之一。

```
Level 3.0：Wide & Deep / DeepFM
Level 4.0：DIN（注意力机制 + 行为序列）
Level 5.0：实时特征流 + 向量召回(ANN) + 多目标排序
```

**下一步最值得做的两个改进**：

1. **时间衰减**：3个月前的接单记录权重应低于昨天的
   `cf_score += sim × exp(-λ × days_ago)`

2. **Item-CF**：找相似任务而不是相似用户
   "你接过任务A，任务B和A被相同的人接过" → 推荐B（更稳定，任务属性固定）

## 总结：改良版系统的核心改进清单

| 改良项 | 改良前 | 改良后 | 效果 |
|--------|--------|--------|------|
| 标签提取 | 正则split | HanLP名词提取 | 噪声大幅减少 |
| abilityScore | 含停用词 | 过滤停用词+小写归一 | 匹配精度提升 |
| categoryScore 未命中 | 0.35 | 0.20 | 区分度增强 |
| amountScore 衰减基准 | 边界值 | 区间宽度 | 修复边界除零bug |
| distanceScore 方向 | 有bug | 修正 | 距离越近分越高 |
| deliveryScore 未命中 | 0.50 | 0.40 | 偏好信号更强 |
| urgencyScore 方向 | 有bug（越近越低）| 修正 | 临近截止分更高 |
