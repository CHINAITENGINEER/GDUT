# 推荐算法学习笔记

> 结合「校园任务接单平台」的真实代码，从零讲透推荐系统的核心思路。

---

## 第一章：推荐系统是什么？解决什么问题？

### 1.1 一句话定义

> **推荐系统 = 在海量候选中，替用户找到「他最可能感兴趣」的那几个。**

没有推荐系统时，用户面对 1000 条任务，要自己翻页筛选，体验很差。
有了推荐系统，系统主动把最合适的 10 条推到他面前。

### 1.2 推荐系统的三要素

```
用户画像（User Profile）
    ↓ 描述「用户是谁、喜欢什么」
候选集合（Candidate Pool）
    ↓ 描述「有哪些东西可以被推荐」
匹配 & 排序（Matching & Ranking）
    ↓ 计算每个候选与用户的相关性得分，排序后返回 Top-K
```

在我们的系统里：
- **用户画像** = `User` 表里的 abilityTags、preferredCategories、金额区间、距离、交付偏好
- **候选集合** = 状态为「待接单」的 Task 列表
- **匹配 & 排序** = `RecommendationServiceImpl` 里的打分逻辑

## 第二章：召回 → 排序，工业界的两阶段范式

工业界（抖音、淘宝）的推荐系统通常分两阶段：

```
全量候选（千万级）
    ↓  【召回 Recall】：粗筛，快速缩小到千级
召回候选（千级）
    ↓  【排序 Ranking】：精排，对每个候选精细打分
最终推荐（十级）
```

我们的系统是「小型单阶段」版本，直接对 300 条候选精排。
但思路是一样的：**先过滤（召回），再打分（排序）**。

### 2.1 我们的召回层：硬过滤（receivable）

```java
// 过滤规则：
// 1. 任务已过截止时间 → 排除
// 2. 金额低于 minAcceptAmount 或高于 maxAcceptAmount → 排除  
// 3. 线下任务且距离超过 maxDistanceKm → 排除
private boolean receivable(Task task, User user) { ... }
```

这叫 **硬约束过滤**：不满足就直接排除，不参与打分。
这样可以大幅减少后续排序的计算量。

### 2.2 我们的排序层：五维加权打分

通过了硬过滤的任务，进入精排阶段。

## 第三章：多维度加权评分模型（核心）

### 3.1 基本公式

$$
\text{score} = \sum_{k} w_k \cdot s_k + \text{bonus}
$$

其中：
- $w_k$ = 第 $k$ 个维度的权重（所有权重之和 = 1）
- $s_k$ = 第 $k$ 个维度的子得分（值域 [0, 1]）
- $\text{bonus}$ = 额外奖励项（时效、信誉、等级匹配）

这是推荐系统里最经典的 **线性加权模型（Linear Weighted Model）**，
简单、可解释、易调参，非常适合冷启动阶段。

### 3.2 五个维度详解

#### 维度1：能力标签匹配（ability） —— Jaccard 相似度

**问题**：如何衡量「用户的技能」和「任务要求的技能」有多匹配？

**方法**：Jaccard 相似度

$$
J(A, B) = \frac{|A \cap B|}{|A \cup B|}
$$

- $A$ = 任务标签集合（从标题+描述提取关键词）
- $B$ = 用户技能标签集合
- 交集越大、并集越小，相似度越高

```
任务标签: {Python, 数据分析, Excel}
用户标签: {Python, 机器学习, Excel}
交集: {Python, Excel} → 大小 = 2
并集: {Python, 数据分析, Excel, 机器学习} → 大小 = 4
Jaccard = 2/4 = 0.5
```

映射到 [0.20, 1.00]：$s_{ability} = 0.20 + 0.80 \times J$

> **为什么不直接用 0~1？**
> 因为即使完全不匹配（J=0），这个任务也有 0.20 的基础分，
> 避免能力维度的权重被「浪费」（对结果没有区分度）。

**关键细节：停用词过滤**

从标题「帮我整理一下数据」提取词：
- 不过滤：{帮, 我, 整理, 一, 下, 数据} ← 噪声很多
- 过滤停用词后：{整理, 数据} ← 干净

停用词（「的」「了」「是」「在」等）对匹配没有意义，必须过滤。

---

#### 维度2：分类偏好匹配（category） —— 二值映射

最简单的维度：命中偏好分类 → 1.0，否则 → 0.20，画像为空 → 0.50。

```
用户偏好分类: [编程, 资料整理]
任务分类: 编程 → 命中 → s_category = 1.0
任务分类: 代取快递 → 未命中 → s_category = 0.20
```

> **为什么未命中给 0.20 而不是 0？**
> 给 0 太极端，会导致任何非偏好分类的任务永远排在最后，
> 即使它在其他维度非常匹配。0.20 是一个「软惩罚」。

---

#### 维度3：金额区间匹配（amount） —— 梯形函数

**问题**：用户设定了「可接受金额 [50, 200]」，任务金额 80 元，得分多少？

**梯形函数**：

```
得分
1.0 |          ___________
    |        /             \
    |       /               \
0.1 |______/                 \______
    0    min                max    金额
```

- 金额在 [min, max] 内 → 1.0（满分）
- 金额偏低（< min）→ 按偏离幅度线性衰减，最低 0.10
- 金额偏高（> max）→ 同样线性衰减，最低 0.10

**关键修复点**：衰减基准要用「区间宽度 span」，而不是边界值本身。

```
错误做法：decay = (min - amt) / min
  → 当 min = 5 时，amt = 0，decay = 1，得分 = 0，但其实只差 5 元

正确做法：decay = (min - amt) / span  (span = max - min)
  → 用区间宽度衡量偏离程度，更合理
```

---

#### 维度4：距离匹配（distance） —— 线性衰减

- 线上任务：不需要出行，固定 1.0
- 线下任务 dist=0：就在附近，1.0
- 线下任务 dist=maxDist：恰好在偏好边界，0.60
- 线下任务 dist > maxDist：超出偏好，线性衰减到 0.10

$$
s_{distance} = \begin{cases}
1.0 & \text{线上任务} \\
1.0 - 0.4 \times \frac{dist}{maxDist} & \text{dist} \leq \text{maxDist} \\
\max(0.10,\ 0.60 - 0.50 \times \frac{dist - maxDist}{maxDist}) & \text{dist} > \text{maxDist}
\end{cases}
$$

> **常见 bug 演示**：原始代码写的是 `ratio = 1 - dist/max`，
> 然后 `score = 0.60 + 0.40 * ratio`。
> 当 dist=0 时 ratio=1，score=1.0 ✓
> 当 dist=max 时 ratio=0，score=0.60 ✓
> 看起来对，但注意：**dist 越大，ratio 越小，score 越小**，方向是对的。
> 实际的 bug 是没有正确处理 dist=0 的返回值（原来是 0.80，不是 1.0）。

---

#### 维度5：交付方式偏好（delivery） —— 二值映射

与分类偏好类似：命中 → 1.0，未命中 → 0.40，未设置 → 0.70。

### 3.3 三项奖励加成

主分之外，还有三个 bonus 项，体现「任务本身的质量」：

#### 时效性奖励（urgencyBonus）

截止日期越近，优先级越高。三段式设计：

```
剩余时间        奖励范围      原因
0 ~ 24h    →  [0.07, 0.10]  非常紧迫，强力推荐
24h ~ 72h  →  [0.02, 0.07)  中等紧迫
72h ~ 168h →  [0,    0.02)  低优先级
> 168h    →  0             不紧迫，无奖励
```

#### 发布者信誉奖励（creditBonus）

```
creditBonus = 0.05 × (credit - baseline) / (max - baseline)
```

信誉越高的发布者，任务越可靠，给予额外加分。

#### 等级匹配奖励（levelBonus）

用户等级对应一个「舒适金额区间」，任务金额落在区间内给满分奖励，
引导用户接「适合自己段位」的任务，避免新手接高难度任务翻车。

## 第四章：权重的秘密——自适应学习

### 4.1 权重是什么？

初始权重：

```
ability:  0.30  ← 技能匹配最重要
category: 0.25
amount:   0.20
distance: 0.15
delivery: 0.10
```

但每个用户的偏好不同，有人最在意金额，有人最在意距离。
**自适应学习的目标：根据用户的真实行为，自动调整这些权重。**

### 4.2 什么时候更新权重？

用户完成一笔订单并结算后，触发权重更新。
这是一种 **隐式反馈（Implicit Feedback）**：
- 用户接了这个任务 → 说明他认可这个任务的某些维度
- 通过分析这个任务在各维度的得分，推断哪些维度对他更重要

### 4.3 梯度下降的思想

**梯度（Gradient）** = 告诉我们权重应该往哪个方向调整、调多少。

核心公式：

$$
w_k^{new} = w_k^{old} + \eta \times g_k
$$

- $\eta$ = 学习率（Learning Rate），我们设为 0.04，控制每次更新幅度
- $g_k$ = 第 k 个维度的梯度信号

**如何计算梯度？**

以子分 0.5 为中心线：
- 子分 > 0.5 → 正梯度（这个维度表现好，增大权重）
- 子分 < 0.5 → 负梯度（这个维度表现差，减小权重）

```
梯度 = (子分 - 0.5) × 2    ← 归一化到 [-1, 1]
```

**举例**：

```
用户接了一个任务：
  ability  子分 = 0.90  → 梯度 = +0.80  → ability 权重增加
  category 子分 = 1.00  → 梯度 = +1.00  → category 权重增加
  amount   子分 = 0.30  → 梯度 = -0.40  → amount 权重减少
  distance 子分 = 0.80  → 梯度 = +0.60  → distance 权重增加
  delivery 子分 = 0.40  → 梯度 = -0.40  → delivery 权重减少

结论：这个用户更在意技能匹配和分类，不太在意金额和交付方式
```

### 4.4 二值维度的特殊处理

category 和 delivery 是二值型（只有命中/未命中），不能简单套用连续型的梯度公式：

- 命中(1.0)  → 梯度 = +1.0
- 未命中(0.20) → 梯度 = -0.60（而不是 -0.80）

为什么减小未命中的负梯度？
> 用户接了一个非偏好分类的任务，**不代表他以后不接这类任务**，
> 可能只是临时需要。用小的负梯度防止过度惩罚。

### 4.5 归一化（Normalize）

更新后权重之和可能不等于 1，需要归一化：

$$
w_k = \frac{w_k}{\sum_j w_j}
$$

同时 clamp 到 [0.05, 0.55]，防止某个维度权重趋近 0 或 1（太极端）。

## 第五章：冷启动问题（Cold Start）

### 5.1 什么是冷启动？

新用户没有任何画像，无法用个性化算法推荐，这就是「冷启动问题」。

### 5.2 我们的解法：基于热度的兜底策略

```
冷启动推荐 = 最新发布的待接单任务
           + receivable 过滤（排除过期/不合条件）
           + urgencyBonus + creditBonus 排序
```

### 5.3 工业界的冷启动方案（扩展阅读）

| 方案 | 思路 |
|------|------|
| 基于热度 | 推热门内容，我们用的就是这个 |
| 基于内容 | 让用户填写偏好标签（我们的「画像」页面） |
| 基于协同过滤 | 找相似用户，推他们喜欢的内容 |
| 基于物品相似 | 找相似任务，推和历史接单相似的任务 |

## 第六章：可解释性（Explainability）

### 6.1 为什么推荐系统需要可解释？

用户看到「技能高度契合 · 佣金在你的期望区间内 · 任务距离较近」，
比只看到分数 0.87 更有信任感，更愿意点击接单。

### 6.2 我们的实现：规则拼接

```java
if (sCategory == 1.0)       reasons.add("符合你的偏好分类");
if (sAbility >= 0.75)       reasons.add("技能高度契合");
if (sAmount  >= 0.90)       reasons.add("佣金在你的期望区间内");
if (urgencyBonus >= 0.07)   reasons.add("截止日期临近，尽快接单");
// 取最强的前 3 条
```

这是最简单的可解释方案：**基于阈值的规则匹配**。
工业界更复杂的方案包括 SHAP 值、注意力权重可视化等。

## 第七章：算法演进路线图

```
Level 1（我们现在）: 多维度加权线性模型
    ↓ 加入用户行为数据（点击、接单、评分）
Level 2: 矩阵分解（Matrix Factorization）/ 协同过滤
    ↓ 加入神经网络
Level 3: Wide & Deep / DeepFM（低阶 + 高阶特征交叉）
    ↓ 加入序列建模
Level 4: DIN（Deep Interest Network）/ 时序 Transformer
    ↓ 大规模工程化
Level 5: 实时特征流 + 向量召回 + 多目标排序
```

### 各级别核心思想

**Level 2 — 协同过滤**
```
「和你相似的用户都接了任务 A」→ 推荐 A 给你
「你接过的任务和任务 B 很像」→ 推荐 B 给你
```

**Level 3 — Wide & Deep**
```
Wide 部分：线性模型，记忆历史规律
Deep 部分：神经网络，泛化到未见过的特征组合
两者结合，兼顾记忆和泛化
```

**Level 4 — DIN（注意力机制）**
```
用户历史行为序列中，不同行为对当前候选的重要性不同
注意力机制自动学习：「历史接过编程任务」对「当前编程候选」贡献更大
```

### 作业3：实现简单的协同过滤（挑战）

思路：
1. 统计所有用户的接单历史
2. 计算用户之间的相似度（余弦相似度）
3. 找到与当前用户最相似的 Top-K 用户
4. 推荐他们接过但当前用户没接过的任务

$$
\text{sim}(u, v) = \frac{\vec{u} \cdot \vec{v}}{|\vec{u}| \cdot |\vec{v}|}
$$

其中 $\vec{u}$ 是用户 u 的接单向量（每个任务对应一维，接过为1，没接为0）。
