# 校园任务接单平台 — 前后端编码规范

> 适用范围：全体开发成员（Java后端 + Vue前端），所有模块均须遵守。

---

## 一、通用规范

### 1.1 命名规范

| 场景 | 规范 | 示例 |
|------|------|------|
| Java类名 | UpperCamelCase | `TaskServiceImpl` |
| Java方法/变量 | lowerCamelCase | `getUserById` |
| Java常量 | UPPER_SNAKE_CASE | `MAX_AMOUNT` |
| 数据库表名 | lower_snake_case（单数） | `task`、`fee_config` |
| 数据库字段名 | lower_snake_case | `created_at`、`publisher_id` |
| Vue组件名 | UpperCamelCase | `TaskCard.vue` |
| Vue变量/方法 | lowerCamelCase | `taskList`、`fetchTaskList` |
| CSS类名 | kebab-case | `.task-card`、`.fee-config` |
| API接口路径 | 全小写 kebab-case | `/api/fee-config`、`/api/my-tasks` |
| Redis Key | 冒号分隔，小写 | `token:{userId}`、`task:detail:{id}` |

### 1.2 注释规范

- **Java**：公共方法必须写 Javadoc（`/** */`），复杂逻辑写行内注释（`//`）。
- **Vue**：组件 `props` 和复杂计算属性必须写注释，业务逻辑超过10行须加说明。
- **SQL**：`Mapper.xml` 中复杂SQL须注释说明查询意图。
- 禁止提交无意义注释（如 `// test`、`// TODO 删掉`）。

### 1.3 文件编码

- 所有文件统一使用 **UTF-8**（无BOM）编码。
- 换行符统一使用 **LF**（Unix风格），在 `.gitattributes` 中配置 `* text=auto eol=lf`。

## 二、时间规范（重点）

### 2.1 核心原则：前后端统一使用时间戳传输

| 场景 | 规范 |
|------|------|
| **接口传输** | 所有时间字段一律使用 **Unix时间戳（毫秒，Long类型）** |
| **数据库存储** | 使用 `DATETIME` 类型存储，由后端完成与时间戳的转换 |
| **前端展示** | 前端接收时间戳后，使用工具函数格式化为可读字符串 |
| **前端传参** | 前端提交时间参数时（如截止时间），传 **毫秒时间戳** |
| **时区** | 统一使用 **UTC+8（北京时间）**，后端配置 `spring.jackson.time-zone=Asia/Shanghai` |

### 2.2 后端时间处理

**Jackson全局配置**（`application.yml`）：

```yaml
spring:
  jackson:
    time-zone: Asia/Shanghai
    date-format: timestamp          # 序列化为毫秒时间戳
    serialization:
      write-dates-as-timestamps: true
```

**实体类字段**：使用 `LocalDateTime`，Jackson自动序列化为毫秒时间戳。

```java
// 正确：实体类使用 LocalDateTime
private LocalDateTime createdAt;
private LocalDateTime deadline;

// 禁止：不要在实体类中使用 String 存储时间
// private String createdAt;  ❌
```

**DTO接收前端时间参数**：用 `Long` 接收时间戳，Service层转换为 `LocalDateTime`。

```java
// DTO
public class TaskPublishDTO {
    @NotNull(message = "截止时间不能为空")
    private Long deadline;   // 前端传入毫秒时间戳
}

// Service层转换
LocalDateTime deadlineTime = LocalDateTime.ofInstant(
    Instant.ofEpochMilli(dto.getDeadline()),
    ZoneId.of("Asia/Shanghai")
);
```

**VO返回前端时间**：直接使用 `Long` 返回时间戳。

```java
// VO
public class TaskVO {
    private Long createdAt;    // 毫秒时间戳
    private Long deadline;     // 毫秒时间戳
    private Long updatedAt;    // 毫秒时间戳
}

// Service层转换
vo.setCreatedAt(task.getCreatedAt()
    .atZone(ZoneId.of("Asia/Shanghai"))
    .toInstant().toEpochMilli());
```

### 2.3 前端时间处理

**统一格式化工具函数**（`src/utils/format.ts`）：

```typescript
import dayjs from 'dayjs'

/**
 * 毫秒时间戳 → 日期时间字符串
 * @example formatTime(1709280000000) → '2024-03-01 12:00:00'
 */
export function formatTime(ts: number | null | undefined): string {
  if (!ts) return '-'
  return dayjs(ts).format('YYYY-MM-DD HH:mm:ss')
}

/**
 * 毫秒时间戳 → 日期字符串
 * @example formatDate(1709280000000) → '2024-03-01'
 */
export function formatDate(ts: number | null | undefined): string {
  if (!ts) return '-'
  return dayjs(ts).format('YYYY-MM-DD')
}

/**
 * 毫秒时间戳 → 相对时间
 * @example formatRelative(ts) → '3小时前'
 */
export function formatRelative(ts: number | null | undefined): string {
  if (!ts) return '-'
  return dayjs(ts).fromNow()
}

/**
 * Date对象/字符串 → 毫秒时间戳（提交时使用）
 * @example toTimestamp(new Date()) → 1709280000000
 */
export function toTimestamp(date: Date | string | null): number | null {
  if (!date) return null
  return dayjs(date).valueOf()
}
```

**前端使用示例**：

```vue
<template>
  <!-- 展示时调用格式化函数，绝不直接渲染时间戳 -->
  <span>{{ formatTime(task.createdAt) }}</span>
  <span>{{ formatRelative(task.deadline) }}</span>
</template>

<script setup lang="ts">
import { formatTime, formatRelative, toTimestamp } from '@/utils/format'

// 提交表单时转换为时间戳
const handleSubmit = () => {
  form.deadline = toTimestamp(pickerValue.value)  // 传给后端
}
</script>
```

> **禁止行为**：
> - ❌ 前端直接展示原始时间戳数字
> - ❌ 后端接口返回 `"2024-03-01 12:00:00"` 格式字符串
> - ❌ 使用 `new Date().getTime() / 1000`（秒级时间戳），统一用**毫秒**

## 三、接口规范

### 3.1 统一返回体

所有接口必须使用统一返回体 `R<T>`，禁止直接返回裸数据或抛出未处理异常。

```java
// 返回体结构
@Data
public class R<T> {
    private Integer code;    // 业务状态码
    private String message;  // 提示信息
    private T data;          // 业务数据
    private Long timestamp;  // 服务器时间戳（毫秒），便于前端校时

    public static <T> R<T> ok(T data) {
        return new R<>(200, "success", data, System.currentTimeMillis());
    }

    public static <T> R<T> fail(Integer code, String message) {
        return new R<>(code, message, null, System.currentTimeMillis());
    }
}
```

**标准返回示例**：

```json
{
  "code": 200,
  "message": "success",
  "timestamp": 1709280000000,
  "data": {
    "id": 1234567890,
    "title": "代取快递",
    "amount": 15.00,
    "createdAt": 1709280000000,
    "deadline": 1709366400000
  }
}
```

### 3.2 业务状态码

| 状态码 | 含义 |
|--------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未登录/Token无效 |
| 403 | 无权限（角色不匹配） |
| 404 | 资源不存在 |
| 409 | 业务冲突（如重复接单） |
| 500 | 服务器内部错误 |

### 3.3 分页接口规范

**请求参数（统一）**：

```json
{
  "page": 1,
  "pageSize": 10
}
```

**返回结构**：

```json
{
  "code": 200,
  "data": {
    "records": [],
    "total": 100,
    "page": 1,
    "pageSize": 10,
    "totalPages": 10
  }
}
```

### 3.4 接口参数校验

- 后端所有 DTO 必须使用 JSR-303 注解校验（`@NotNull`、`@NotBlank`、`@Range` 等）。
- Controller 方法参数加 `@Valid`，全局异常处理器捕获 `MethodArgumentNotValidException`。
- 前端提交前必须做表单校验，不依赖后端兜底。

```java
// 示例
public class TaskPublishDTO {
    @NotBlank(message = "任务标题不能为空")
    @Length(max = 100, message = "标题最长100字")
    private String title;

    @NotNull(message = "任务金额不能为空")
    @DecimalMin(value = "1.00", message = "金额最低1元")
    @DecimalMax(value = "500.00", message = "金额最高500元")
    private BigDecimal amount;

    @NotNull(message = "截止时间不能为空")
    private Long deadline;  // 毫秒时间戳
}
```

## 四、后端编码规范

### 4.1 项目结构规范

- 每个模块下严格按 `controller` / `service` / `service/impl` / `mapper` / `entity` / `dto` / `vo` 分包。
- `entity` 只做数据库映射，不含业务逻辑。
- `dto`（Data Transfer Object）：接收前端入参。
- `vo`（View Object）：返回给前端的数据，按需裁剪字段（**禁止直接返回 entity**）。

### 4.2 Service层规范

- 业务逻辑全部写在 Service 实现类中，Controller 只做参数接收和结果返回。
- 涉及多表写操作必须加 `@Transactional`，只读操作加 `@Transactional(readOnly = true)`。
- Service 中操作 Redis 缓存统一封装，不在 Controller 中直接操作 `redisTemplate`。

```java
// 正确：Controller 保持简洁
@GetMapping("/{id}")
public R<TaskVO> getDetail(@PathVariable Long id) {
    return R.ok(taskService.getTaskDetail(id));
}

// 禁止：Controller 中写业务判断
// if (task.getStatus() == 0) { ... }  ❌
```

### 4.3 MyBatis-Plus 规范

- 简单CRUD优先使用 MyBatis-Plus 内置方法，复杂多表查询写 `Mapper.xml`。
- 禁止在循环中执行数据库查询（N+1问题），改用批量查询。
- 分页统一使用 MyBatis-Plus `Page<T>` + `IPage<T>`。

```java
// 正确：分页查询
Page<Task> page = new Page<>(dto.getPage(), dto.getPageSize());
LambdaQueryWrapper<Task> wrapper = new LambdaQueryWrapper<Task>()
    .eq(Task::getStatus, 2)
    .orderByDesc(Task::getCreatedAt);
return taskMapper.selectPage(page, wrapper);
```

### 4.4 异常处理规范

- 业务异常统一抛出 `BusinessException(code, message)`，由全局异常处理器捕获并返回标准 `R`。
- 禁止在 Service 中 `try-catch` 后吞掉异常（打日志后必须重新抛出或处理）。
- 禁止将异常堆栈信息直接返回给前端。

```java
// 正确：抛出业务异常
if (task.getStatus() != 2) {
    throw new BusinessException(409, "任务状态不允许接单");
}

// 禁止：返回null或吞掉异常
// return null;  ❌
// catch(Exception e) { log.error(e); }  ❌
```

### 4.5 日志规范

- 使用 `@Slf4j` 注解，禁止使用 `System.out.println`。
- 日志级别：`DEBUG` 调试信息、`INFO` 关键业务节点、`WARN` 潜在问题、`ERROR` 异常。
- 禁止在日志中打印用户密码、Token等敏感信息。

```java
// 正确
log.info("用户[{}]发布任务[{}]，金额:{}", userId, taskId, amount);
log.error("任务结算失败，taskId:{}", taskId, e);

// 禁止
System.out.println("发布成功");  // ❌
log.info("用户密码:{}", password);  // ❌
```

### 4.6 金额处理规范

- 所有金额字段使用 `BigDecimal`，**禁止使用 `double` / `float`**（精度丢失问题）。
- 金额运算使用 `BigDecimal` 方法（`add` / `subtract` / `multiply`），指定精度和舍入模式。
- 数据库字段使用 `DECIMAL(10,2)`。

```java
// 正确
BigDecimal feeAmount = amount.multiply(feeRate)
    .setScale(2, RoundingMode.HALF_UP);
BigDecimal realAmount = amount.subtract(feeAmount);

// 禁止
double fee = amount * 0.05;  // ❌ 浮点精度丢失
```

## 五、前端编码规范

### 5.1 Vue3 组件规范

- 统一使用 `<script setup lang="ts">` 组合式API写法。
- 组件必须定义 `props` 类型，使用 `defineProps<{}>()` 泛型写法。
- 禁止在 `template` 中写复杂逻辑，抽离为 `computed` 或方法。

```vue
<!-- 正确 -->
<script setup lang="ts">
const props = defineProps<{
  taskId: number
  status: number
}>()

const statusText = computed(() => {
  const map: Record<number, string> = {
    1: '待支付', 2: '发布中', 3: '已接单', 4: '已完成', 5: '已结算'
  }
  return map[props.status] ?? '未知'
})
</script>
```

### 5.2 Axios 请求规范

- 所有接口调用统一封装在 `src/api/` 目录下，**禁止在组件中直接写 axios 调用**。
- 请求拦截器自动注入 Token，响应拦截器统一处理 401（跳登录页）和业务错误码弹提示。

```typescript
// src/utils/request.ts
const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 10000
})

// 请求拦截：注入Token
request.interceptors.request.use(config => {
  const token = useUserStore().token
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})

// 响应拦截：统一处理错误
request.interceptors.response.use(
  res => {
    if (res.data.code !== 200) {
      ElMessage.error(res.data.message)
      return Promise.reject(res.data)
    }
    return res.data.data  // 直接返回 data 字段
  },
  err => {
    if (err.response?.status === 401) router.push('/login')
    return Promise.reject(err)
  }
)
```

### 5.3 状态管理规范

- Pinia Store 按功能模块拆分，禁止建单一全局 Store 堆积所有状态。
- Store 中的异步操作写在 `actions` 中，不在组件里直接调 API 再手动赋值给 Store。

```typescript
// src/stores/user.ts
export const useUserStore = defineStore('user', () => {
  const token = ref<string>(localStorage.getItem('token') ?? '')
  const userInfo = ref<UserInfo | null>(null)

  async function login(data: LoginDTO) {
    const res = await authApi.login(data)
    token.value = res.token
    localStorage.setItem('token', res.token)
  }

  function logout() {
    token.value = ''
    userInfo.value = null
    localStorage.removeItem('token')
  }

  return { token, userInfo, login, logout }
})
```

### 5.4 环境变量规范

- 所有环境相关配置写在 `.env` 文件中，不硬编码在代码里。
- 变量名以 `VITE_` 开头才能在前端代码中访问。

```bash
# .env.development
VITE_API_BASE_URL=http://localhost:8080

# .env.production
VITE_API_BASE_URL=https://your-domain.com
```

### 5.5 禁止行为清单

| 禁止 | 原因 |
|------|------|
| `any` 类型滥用 | 失去TypeScript类型保护 |
| 直接操作 `localStorage`（Store外） | 状态不同步 |
| 在 `template` 中直接写 `console.log` | 调试代码不应提交 |
| 金额展示不做格式化 | 如 `15` 应展示为 `¥15.00` |
| 时间戳直接渲染 | 应调用 `formatTime()` |
| 硬编码 API 地址 | 应使用 `VITE_API_BASE_URL` |

## 六、Git 提交规范

### 6.1 分支规范

```
main          # 主分支，仅合并稳定版本
dev           # 开发主分支，日常合并
feature/xxx   # 功能分支，如 feature/task-module
bugfix/xxx    # Bug修复分支
```

### 6.2 Commit Message 格式

```
<type>(<scope>): <subject>

类型（type）：
  feat     新功能
  fix      Bug修复
  docs     文档变更
  style    代码格式（不影响功能）
  refactor 重构
  test     测试
  chore    构建/工具变更

示例：
feat(task): 新增任务发布接口
fix(payment): 修复手续费计算精度问题
docs: 更新接口文档
```

### 6.3 禁止提交内容

- 密码、密钥、Token等敏感信息（`.env.local` 加入 `.gitignore`）
- `node_modules/`、编译产物 `target/` `dist/`
- 调试用的 `console.log`、`System.out.println`
- 未解决的冲突标记（`<<<<<<`）

## 七、规范速查表

| 场景 | 规范要点 |
|------|----------|
| 时间传输 | **毫秒时间戳（Long）**，前端用 `formatTime()` 展示 |
| 金额传输 | `BigDecimal`，前端展示加 `¥` 前缀保留2位小数 |
| 接口返回 | 统一 `R<T>` 结构，含 `code`/`message`/`data`/`timestamp` |
| 分页参数 | 请求 `page`+`pageSize`，返回 `records`+`total`+`totalPages` |
| 密码传输 | 前端MD5一次后传输，后端 SHA256+盐 存储 |
| 错误处理 | 后端抛 `BusinessException`，前端响应拦截器统一弹提示 |
| 缓存Key | 冒号分隔小写，如 `task:detail:{id}` |
| ID类型 | 后端 `Long`，前端 `string`（防止JS大数精度丢失） |

> **关于ID精度问题**：Java `Long` 类型最大值超过 JavaScript `Number` 安全整数范围，后端返回雪花ID时需序列化为字符串。
> 配置方式：在 `Jackson` 中注册 `Long` → `String` 序列化模块，或在实体类 `id` 字段加 `@JsonSerialize(using = ToStringSerializer.class)`。
